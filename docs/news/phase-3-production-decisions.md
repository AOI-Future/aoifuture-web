# AOIFUTURE News Phase 3 — 本番メタデータ／フィード判断メモ

Status: **OWNER DECISION REQUIRED — 実装・公開は未承認**

## 先に選ぶもの

メタデータとフィードは別々に選ぶ。ここを一つの「ニュース対応」として決めると、検索のための記述と配信のための運用が混ざる。

- **メタデータの推奨:** `M2 編集ダイジェスト／コレクション`
- **フィードの推奨:** `F1 Edition RSS`
- **組み合わせの意図:** AOIFUTURE Newsを「記事の量産媒体」ではなく、一次情報を有限のEditionに編む編集面として機械にも伝える。配信も一日分を一つのまとまりとして届ける。

この推奨は実装指示ではない。OwnerがM1〜M3とF0〜F3をそれぞれ選んだ後に、公開フィールド、訂正テスト、robots変更、フィード出力を別タスクで設計する。

## 判断の前提

Debug GateはPhase 3候補 `987315f41a3e72fe33e99b4c9926cf60dadd7eed` をPASSとした。69件のNewsテスト、契約検証、build、13件のE2E、14件の密度／Detourシナリオが通り、生成されたNews HTMLに禁止情報の検出はなかった。

ただし、現状は非本番サンプルである。4つのNewsルートには、ページタイトル、description、`https://aoifuture.com` のcanonical、`noindex, nofollow` がある。一方、OGP、News向け構造化データ、RSS/Atomはまだない。したがって、この文書が比較するのは「現在の公開能力」ではなく、Owner選択後にどこまで作るか、である。

## 1. メタデータを選ぶ

### M1 — 最小限のWebページメタデータ

ページ固有のtitle、description、canonicalを保ち、本番化時にrobots方針を切り替える。必要なら共有用の基本メタデータを足すが、EditionやSignalをニュース発行物として構造化しない。

| 判断軸 | 内容 |
| --- | --- |
| 読者体験 | ブラウザと検索結果からページ名は判別できる。Editionの編集単位や収録Signalの関係は機械には伝わりにくい。 |
| 発見 | 通常のWebページとして発見される余地はあるが、日付付きEdition、一覧、更新の意味を明示しない。 |
| 訂正 | 可視ページの訂正表示が正本。メタデータ側で訂正日時や変更理由を追わせないため、外部キャッシュとのずれを見つけにくい。 |
| 著作権・プライバシー | 公開する記述が最少で、漏えい面も小さい。外部ソース本文や私的編集情報を複製しない境界を守りやすい。 |
| 運用コスト | 最小。ページのtitle、description、canonical、robotsをリリース時に確認すればよい。 |
| 実装範囲 | NewsLayoutと各ルートのhead、生成HTML検証、robots／sitemapの本番方針。 |

**向く選択:** まず検索可能なWebページだけを安全に出し、編集面としての機械可読性は後日に送る場合。

### M2 — 編集ダイジェスト／コレクション [推奨]

Editionを日付付きの編集コレクションとして表し、ページ、Edition、収録項目の関係を機械可読にする。SignalをAOIFUTURE執筆の独立ニュース記事とは扱わない。

| 判断軸 | 内容 |
| --- | --- |
| 読者体験 | 検索や共有から「この日の有限な選集」へ入りやすい。一次情報へ進む現在の導線と、Edition全体を読む体験を両立しやすい。 |
| 発見 | Editionの日付、見出し、説明、収録順、canonicalを一つの編集単位として示せる。Signal単体を検索向けに過剰分解しない。 |
| 訂正 | Editionの更新日時と、公開済みの`corrected_at`／訂正注記を対応させる設計が必要。外部キャッシュが即時更新されるとは限らないため、ページ上の可視訂正を正本のまま保つ。 |
| 著作権・プライバシー | 構造化データには公開審査済みのAOIFUTURE見出し、短い説明、日付、公開URLだけを使う。ソース本文、RSS本文、receipt、内部スコア、私的メモは載せない。M1より公開面は増えるが、境界は限定できる。 |
| 運用コスト | 中。Edition生成、順序、日時、訂正状態とhead出力の整合を継続テストする。 |
| 実装範囲 | Edition用の構造化データと共有メタデータ、公開フィールドのallowlist、訂正／取り下げ時の出力テスト、生成HTMLの漏えい検査。 |

**推奨理由:** 製品の中心は「一つずつの記事」ではなく、一次情報を並べた有限のDaily Editionである。M2はその形を崩さず、M3ほどの発行者運用を先に背負わない。

### M3 — フルNews Publisherメタデータ

AOIFUTUREをニュース発行主体として定義し、著者・発行者・ロゴ・公開／更新時刻・記事単位などを広く構造化する。Signalをどの型で表すかも含め、発行責任のモデルを先に固定する選択である。

| 判断軸 | 内容 |
| --- | --- |
| 読者体験 | 記事単位の検索表示や共有を狙いやすい反面、読者をEditionではなくSignal断片へ直接入れやすい。一次情報への案内カードをAOIFUTURE自身の記事と誤認させない設計が要る。 |
| 発見 | 三案で最も細かく記述できる。ただし、細かい記述がそのまま露出向上を保証するわけではない。 |
| 訂正 | Signalごとの公開時刻、更新時刻、訂正、取り下げ、canonical、識別子を同期する必要がある。古い構造化データや外部キャッシュとの差分監視も運用対象になる。 |
| 著作権・プライバシー | Signal単位で説明や出典情報を広く配るほど、ソース表現の再配布と内部情報混入の確認箇所が増える。発行者・著者情報も公開範囲を明示して管理する必要がある。 |
| 運用コスト | 最大。記事型の意味、著者責任、ロゴ、訂正SLA、検証、監視まで継続運用になる。 |
| 実装範囲 | 発行者／著者モデル、記事または代替型の設計、Signal単位の構造化出力、訂正・取り下げ・重複URLのテスト、公開後の検索系モニタリング。 |

**いま選ばない理由:** 現在のSignalは一次情報へ送る編集カードであり、AOIFUTURE執筆記事そのものではない。意味のずれを残したまま「ニュース媒体らしい」記述だけを増やすのは早い。

## 2. フィードを選ぶ

フィード本文に外部記事やRSS本文を転載しないことは全案共通とする。公開するなら、公開審査済みのAOIFUTURE文面、AOIFUTURE URL、原典への明示的なリンクだけで組む。

### F1 — Edition RSS：一日一項目 [推奨]

一つのDaily Editionを一つのRSS itemとして配る。項目はEditionの見出し、短い説明、公開日時、AOIFUTUREのEdition URLを中心にし、読む本体はWebの有限なEditionに置く。

| 判断軸 | 内容 |
| --- | --- |
| 読者体験 | 購読通知が一日一つにまとまり、その日の編集全体を見失いにくい。個別Signalだけを素早く拾いたい読者には一手多い。 |
| 発見 | RSSリーダーやフィード連携に日次の入口を作れる。Signal単位の検索・通知性はF2より弱い。 |
| 訂正 | Edition itemの安定IDを保ち、更新日時と訂正表示を更新する。リーダーが更新を再表示しない場合があるため、訂正の正本はEditionページに置く。 |
| 著作権・プライバシー | 配信payloadをEdition見出しと短いAOIFUTURE説明に絞れる。外部本文や内部編集情報を再配布する危険を抑えやすい。 |
| 運用コスト | 低〜中。Edition公開時に一項目を生成し、再生成、訂正、URL安定性を確認する。 |
| 実装範囲 | `/news/feed.xml`、安定GUID、公開／更新日時、短いdescription、feed discovery link、出力allowlist、訂正テスト。 |

**推奨理由:** 日付付きの有限なEditionを製品の単位として守れる。M2とも同じ粒度で噛み合う。

### F2 — Signal feed：Signalごとに一項目

一つのSignalを一つのfeed itemとして配る。

| 判断軸 | 内容 |
| --- | --- |
| 読者体験 | 原典候補を一件ずつ素早く流し込める。ただし、通知量が増え、Edition note、並び、隣接関係という編集価値が薄くなる。 |
| 発見 | Signal単位の購読、保存、自動処理に向く。全体像より個別項目が前へ出る。 |
| 訂正 | Signalごとの安定ID、訂正日時、取り下げ、URL差し替えを配信に反映する。複数Editionに関係するSignalの重複規則も必要になる。 |
| 著作権・プライバシー | 一項目ごとの説明を厚くするほど、ソース本文の代替物になりやすい。公開フィールドのallowlistと文字量の上限を決め、receiptや内部スコアを必ず遮断する。 |
| 運用コスト | 中〜高。項目数に比例して検証、訂正、重複排除、購読者への変化が増える。 |
| 実装範囲 | Signal-to-feed変換、GUID、Editionとの帰属、原典リンク、訂正／取り下げ／重複規則、feed単体の漏えい検査。 |

**向く選択:** Editionよりも、個別Signalを外部のワークフローへ流すことを製品価値として優先する場合。

### F3 — Edition + Active Context activity feed

Daily Editionの公開と、Active Contextの意味ある更新を同じ活動フィードへ流す。

| 判断軸 | 内容 |
| --- | --- |
| 読者体験 | 「今日選んだもの」と「見方が変わったこと」を一か所で追える。一方、日次スナップショットと更新され続けるContextが混ざり、項目の種類を読者が毎回判別する必要がある。 |
| 発見 | AOIFUTURE Newsの二つの時間軸を配信できる。通常のニュース購読より、変更履歴を追う読者や機械処理に向く。 |
| 訂正 | Edition訂正とContext revisionを別種のイベントとして安定IDで扱う。訂正、見解更新、取り下げを混同しない状態遷移が必要になる。 |
| 著作権・プライバシー | 公開payloadと更新頻度が最大になる。Contextの根拠は公開Signal参照だけに限定し、過去の私的メモや収集履歴を履歴化しない。 |
| 運用コスト | 最大。二種類の正本、イベント順序、重複、再送、訂正を監視する。 |
| 実装範囲 | 型付きactivity feed、Edition／Contextイベント、revision ID、順序と再生成規則、購読表示、漏えい／回帰テスト。 |

**いま選ばない理由:** Active Contextの更新運用が実地で固まる前に統合フィードを出すと、何を「更新」として通知するかが不安定になる。

### F0 — まだフィードを出さない

WebのEditionとArchiveだけを公開し、フィードは保留する。

| 判断軸 | 内容 |
| --- | --- |
| 読者体験 | Webでは読めるが、RSS購読や外部ワークフローへの定期入口がない。読者が自分で再訪する必要がある。 |
| 発見 | Web検索と直接導線に限定される。RSSリーダー、個人自動化、フィードディレクトリからの発見は得られない。 |
| 訂正 | 可視ページだけを直せばよく、配信済み項目との不一致は起きない。 |
| 著作権・プライバシー | 追加の配信面がなく、四案で最も露出が小さい。 |
| 運用コスト | 最小。フィード生成、検証、訂正、監視が不要。 |
| 実装範囲 | なし。将来の選択を妨げないよう、公開データ契約だけを安定させる。 |

**向く選択:** 最初のEdition運用と訂正手順をWebだけで観察してから、配信粒度を決めたい場合。

## 3. 6／9／12 Signalの密度判断

6、9、12の全構成は、1440pxと390pxで横overflowがなく、Signalごとに一つの原典アクションを保った。ページネーションも折りたたみもなく、Edition noteとfooterはDOM上・計測上とも残っている。つまり、12件でも「壊れない」。けれど、壊れないことと読みやすいことは同じではない。

| Signal数 | Desktop | Mobile 390 | 判断 |
| ---: | ---: | ---: | --- |
| 6 | 6,485px / 6.49 viewport | 7,041px / 8.34 viewport | 両幅で無理が少ない。通常Editionの基準候補。 |
| 9 | 9,103px / 9.10 viewport | 9,923px / 11.76 viewport | 構造は明瞭。mobileでは走査距離が長い。 |
| 12 | 11,662px / 11.66 viewport | 12,777px / 15.14 viewport | 技術的には有限。二種類のカード反復と、最後の原典／footerまでの距離が明確な疲労点。 |

**推奨:** 6件を通常の出発点、9件を編集上の必要がある日の上限目安とする。12件は、各Signalを残す理由が全体で説明できる場合の例外に寄せる。件数だけで自動的に落とすのではなく、薄いSignalを増やしてEditionを長くしない。

## 4. Detourは「置けるか」ではなく「Editionが良くなるか」で決める

9 Signalの比較では、Detourなしが最も強いsource-to-sourceのリズムを保った。compactを一つ、Signal 3の後へ置いた案は、将来の内容が本当に独立している場合に限り、最も割り込みが小さい。full-widthは割り込みが強い。4つを過剰配置した例は、原典アクションを一つも増やさず、ページをdesktop 10.03 viewport、mobile 13.23 viewportまで延ばした。

ここで固定の「何件目ならDetour可」という規則は作らない。見るのは、whole Editionのバランスである。

別ブロックにする条件は三つだけだ。

1. Source fact、AOI note、Caveat、Edition note、Active Contextのどれとも重ならない情報である。
2. 原典を追う流れを切る以上の、明確な方向づけや意外な接続を生む。
3. mobile、JavaScriptなし、Edition全体の走査で意味が残る。

この条件を満たさないなら、**Detourを消すのではなく、情報を通常のEdition contentへ戻して全部残す。** 現時点の推奨は「別Detourなし」。将来、実データでwhole-Edition balanceが改善したと確認できたときだけ、compact案から再評価する。
